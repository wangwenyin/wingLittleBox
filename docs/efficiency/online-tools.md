# 在线工具

在线工具最有价值的地方，不是“什么都能做”，而是遇到碎任务时能立刻补位。

从这次书签里筛下来，真正适合放进 `efficiency / online-tools` 的，不是数量最多的那批，而是这些打开快、用途稳、能反复进入工作流的工具。

## 我筛选它们的标准

| 标准 | 说明 |
| --- | --- |
| 高频碎任务 | 偶尔不用，但一旦用上通常很急 |
| 即开即用 | 不想为了一个小任务专门开本地软件 |
| 结果直观 | 输入和输出足够清晰，能立刻得到反馈 |
| 适合常驻 | 值得留在收藏夹，而不是临时搜索一次就走 |

## 值得常驻的主力工具

| 类别 | 工具 | 适合场景 | 推荐理由 |
| --- | --- | --- | --- |
| 正则调试 | [Regulex](https://jex.im/regulex/#!flags=&re=%5E(a%7Cb)*%3F%24) | 看复杂正则结构、排查分组关系 | 可视化足够直观，适合把“看不懂的正则”拆开来看 |
| 代码分享 | [Carbon](https://carbon.now.sh/) | 生成代码截图、写文章配图、做分享封面 | 出图快，默认观感就很好，适合内容输出 |
| 流程图 | [ProcessOn](https://www.processon.com/) | 画流程图、结构图、协作脑图 | 对轻量文档协作很友好，比打开重型设计工具更轻 |
| API 协作 | [Apifox](https://apifox.com/) | 接口文档、调试、Mock、联调 | 适合把文档和调试放在同一入口，减少切换 |
| 在线请求调试 | [Postman Web](https://web.postman.co/) | 临时发请求、复现接口、保存示例 | 偶发联调很方便，尤其适合跨设备快速打开 |
| CDN 检索 | [cdnjs](https://cdnjs.com/) | 查前端库 CDN 地址、版本号、引用方式 | 输入库名就能用，适合处理临时引用需求 |
| 图片压缩 | [迅捷图片压缩](https://yasuo.xunjiepdf.com/img/) | 临时压缩配图、博客图片、文档插图 | 不值得为一两张图开本地软件时很好用 |
| 在线代码运行 | [菜鸟 Node.js 在线工具](https://c.runoob.com/compile/22/) | 临时验证小段 JS / Node 逻辑 | 用来试小片段比新建本地文件更省动作 |

## 适合作为补位入口的工具

| 工具 | 更适合什么时候用 | 备注 |
| --- | --- | --- |
| [在线接口文档工具](https://easydoc.net/p/60163813/jHMqaINf) | 需要快速生成或查看轻量接口文档时 | 如果团队已经统一用 Apifox，这类工具可以降级为备用 |
| [coderutil 编程工具导航](https://www.coderutil.com/) | 一时想不起来某个编解码、转换、格式化工具时 | 更像聚合入口，适合兜底，不建议当主力 |

## 不建议继续堆收藏的类型

| 类型 | 原因 |
| --- | --- |
| 纯导航站过多 | 最后会变成“收藏了很多，但还是靠搜索” |
| 和本地工具强重叠的网页工具 | 如果你已经长期重度使用桌面端，网页版本只保留一个备用入口就够了 |
| 私有后台、临时项目链接 | 它们属于工作书签，不适合放进通用提效文章里 |

## 更适合这页的排法

如果长期维护这一页，我更建议按“任务”而不是按“网站名”来记：

| 任务 | 建议保留的入口 |
| --- | --- |
| 看正则 | Regulex |
| 出代码图 | Carbon |
| 画流程 | ProcessOn |
| 联调接口 | Apifox / Postman Web |
| 查 CDN | cdnjs |
| 压图片 | 迅捷图片压缩 |
| 跑小片段 | 菜鸟在线运行 |

## 小结

在线工具不需要多，关键是稳定。

把正则、接口、流程图、代码截图、图片压缩这几类入口固定下来，已经足够覆盖大多数“临时但重复”的工作流空档。

# Visual Studio Code 配置

VS Code 的关键不在于装很多插件，而在于形成稳定、可迁移、低冲突的开发环境。真正高效的编辑器配置，通常不是“功能越多越好”，而是让格式化、跳转、诊断、搜索和调试都保持可预期。

## 配置目标

我更推荐把 VS Code 的配置目标定成下面三件事：

- 跨项目切换时行为稳定
- 团队协作时不制造额外差异
- 常见操作尽量少思考、少手动

## 基础配置方向

最值得先统一的是这些：

- 字体与字号
- 自动保存策略
- 格式化规则
- 终端行为
- 文件排除与搜索排除

这些配置虽然不“炫”，但对日常体验影响最大。

## 插件选择原则

先说原则，再说清单：

- 插件只解决明确问题
- 同类插件不要并存太多
- 项目规则优先于个人偏好
- 不把编辑器变成另一套构建系统

最常见的低效来源，不是功能不够，而是插件互相打架。

## 推荐插件方向

### 通用开发

- ESLint
- Prettier
- EditorConfig
- GitLens

这组基本覆盖了：

- 代码质量提示
- 格式统一
- 编辑器与项目风格对齐
- Git 上下文查看

### Vue / 前端

- Vue - Official
- TypeScript 相关增强
- Tailwind CSS IntelliSense

这里的重点不是“全装”，而是跟当前项目技术栈一致。

### 文档写作

- Markdown 预览增强
- 拼写检查

如果你经常写文档、博客或技术笔记，这类插件比很多花哨能力更实用。

## 最值得统一的工作流

### 格式化由项目接管

编辑器可以触发格式化，但“格式化结果”最好由项目规则决定。

这意味着：

- 项目有 ESLint / Prettier 配置时，以项目为准
- 不要用全局个人设置去覆盖团队规则
- 保存即格式化要建立在规则稳定的前提上

### 搜索和排除项要干净

大项目里搜索慢、结果乱，往往是因为没有配置好排除项。至少应该明确过滤：

- `node_modules`
- 构建产物
- 缓存目录
- 大型自动生成文件

这样全局搜索的信噪比会高很多。

### 终端要成为顺手的入口

如果你频繁在编辑器和外部终端之间来回切，通常说明终端配置还不够顺。

建议统一：

- 默认 shell
- 启动目录行为
- 常用任务触发方式

## 一个更实用的维护思路

把配置分三层看：

### 全局级

只放和个人习惯强相关、且不会破坏项目协作的东西，例如：

- 字体
- 主题
- 光标行为

### 项目级

放与当前仓库强相关的配置，例如：

- 格式化策略
- 语言特定行为
- 推荐插件

### 团队级

通过仓库配置文件统一：

- Lint 规则
- Formatter 规则
- TypeScript / Vue / 构建相关约束

## 建议的工作流

- 项目级优先于全局级配置
- 把格式化规则交给项目统一
- 插件数量保持克制，避免互相冲突
- 定期回收“装了但几乎不用”的插件

## 后续可以继续补的内容

- 常用快捷键
- `settings.json` 样例
- snippet 模板
- 调试配置模板

## 小结

VS Code 的效率，不是靠插件堆出来的，而是靠：

- 清晰的配置分层
- 稳定的格式化与诊断链路
- 干净的搜索和终端体验

这些一旦稳定下来，编辑器就会真正变成“低摩擦工具”。

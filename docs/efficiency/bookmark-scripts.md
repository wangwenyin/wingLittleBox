# 书签脚本

书签脚本适合解决“偶发但重复”的轻量网页操作。它的优势不是能力上限高，而是触发成本很低：不用装扩展、不用打开控制台、不用单独跑脚本。

这类能力用得好，会成为很轻的网页操作工具箱。

## 适合什么场景

书签脚本最适合那些：

- 逻辑简单
- 触发频率不低
- 又不值得为此写完整扩展

的网页操作任务。

常见场景包括：

- 页面样式调试
- 批量展开折叠内容
- 提取页面图片链接
- 快速清理干扰元素
- 辅助测试 DOM 状态

## 一个简单示例

高亮页面中的所有图片：

```js
javascript:(() => {
  document.querySelectorAll('img').forEach((img) => {
    img.style.outline = '2px solid #ff4d4f'
  })
})()
```

这个例子的意义不在于它本身有多复杂，而在于它说明了书签脚本的典型价值：

- 进入页面
- 点一下书签
- 立刻看到结果

## 什么时候它比扩展更合适

如果一个需求只是：

- 临时辅助调试
- 定向清理页面
- 快速读取某类 DOM 信息

那书签脚本通常比装扩展更轻。

但如果需求开始涉及：

- 长期后台运行
- 复杂状态管理
- 跨站点能力

那它通常就不再是合适方案了。

## 写书签脚本时要注意什么

### 保持单一职责

一个书签脚本最好只做一件事，例如：

- 高亮图片
- 展开全部评论
- 拿到所有链接

否则后续很难维护。

### 先在控制台验证

最稳妥的做法是：

1. 先在浏览器控制台跑通
2. 再压成书签脚本
3. 最后补上清晰命名

这样比直接手写 `javascript:` 版本更不容易出错。

### 不要在敏感站点乱跑未知脚本

书签脚本本质上仍然是在当前页面上下文执行 JavaScript，所以：

- 只运行自己理解的脚本
- 不把未知来源脚本直接加进常用书签
- 对登录页、支付页、后台管理页尤其谨慎

## 使用建议

- 只对自己理解的脚本执行
- 不在敏感站点随意运行未知书签脚本
- 把常用脚本配上明确命名和注释
- 按用途分组管理，而不是全部堆在书签栏

## 后续可整理方向

- DOM 调试类
- 页面清洗类
- 抓取辅助类
- 自动化辅助类

## 小结

书签脚本的价值，在于它填补了“太轻，不值得装扩展；太重复，不想每次开控制台”的那段空白。只要把它控制在简单、明确、可理解的范围内，它会是很高效的一类小工具。
